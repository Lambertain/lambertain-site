# Спека: интейк клиента (Лид → Бриф → Договор → Спека → Проект)

> Статус: план. Реализуется инкрементом (модули 1→4). Дизайн и спека проекта делаются **локально** Claude Code
> с дизайн-китом (`.claude/skills/ui-ux-pro-max`) и спек-китом (`decomposeSpec`/kickoff) — портал собирает вход (бриф) и доки.

## 1. Проблема
Сейчас клиент заводится через одну жёсткую страницу «онбординг», привязанную к уже существующему проекту.
Нужно: (а) собирать **бриф ещё до клиента/цены/проекта**, (б) растущую библиотеку страниц-инструкций («конструкторы»),
(в) генерацию договоров/счетов, (г) единый мастер заведения проекта. Дизайн при этом не должен быть «ИИ-дефолтом»
(warm-cream/serif/terracotta) — он рождается из дизайн-кита по брифу.

## 2. Жизненный цикл
0. **Лид + Бриф (без проекта).** Я завожу лида, шлю ссылку на бриф. Лид открывает Mini App и видит **только бриф**.
   Заполнил → мне уведомление.
1. **Оценка + договор.** По брифу понимаю объём → озвучиваю цену → беру реквизиты → формирую договор + счёт.
2. **Дизайн + спека.** Локально из брифа: дизайн-кит → арт-дирекшен + превью; спек-кит → задачи. Автоопределяю,
   что клиенту регистрировать.
3. **Проект.** Создаю проект, привязываю бриф, присваиваю лиду → он становится клиентом с проектом → вторая
   ссылка: гайды-регистрации (фаза 2).

## 3. Состояния пользователя
- **Лид** = пользователь с ролью `client`, **без `projectKey`**, с флагом «бриф ожидается/получен». Проекта ещё нет.
- **Клиент с проектом** = после создания+присвоения проекта.
- Mini App: нет проекта → показываем **только бриф** (не пустой экран задач; пустое состояние ≠ ошибка).

## 4. Модель данных (изменения)
- `briefs` — бриф на уровне лида: `id, lead_tg_id/contact, project_type, payload(jsonb), status(draft|submitted), project_key(nullable), created_at`.
  Привязка `project_key` проставляется позже, при создании проекта.
- `invites` — новый тип **brief-only**: даёт доступ к брифу без проекта (приглашённый = `client`, `project_key=null`).
- `checklist_pages` — библиотека многоразовых страниц-инструкций: `slug, title, blocks(jsonb), phase(1|2)`.
  Каждая страница = набор блоков с тумблерами.
- `project_checklist` — какие страницы/блоки включены для конкретного проекта (+ фаза).
- `legal_entities` — мои юрлица (ФОП-1/ФОП-2/ТОВ) с реквизитами для подстановки.
- `contract_templates` — шаблоны (договор A/B, НДА) с плейсхолдерами `{{...}}`.
- `client_requisites` — реквизиты клиента (на лиде/проекте) для договора/счёта.
- `documents` — сгенерированные доки (договор/НДА/счёт) для скачивания.

## 5. Модули и порядок постройки
1. **Лид + Бриф** (project-less) — ✅ **РЕАЛИЗОВАНО (core)**. Решение проще, чем «Mini App только бриф»: бриф собирается по
   **публичной токен-ссылке** `/brief/<token>` (минимум трения для лида — открыл в браузере, заполнил), без Telegram-авторизации
   на этом этапе. Таблица `briefs`, адаптивная форма (тип проекта → ветка), submit + уведомление в бот, админ-раздел **«Брифы»**
   (создать лида → ссылка, список с ответами). *TODO:* `/api/dev/brief` (пока ответы копирую из админки).
2. **Привязка бриф→проект** — ✅ **РЕАЛИЗОВАНО (core)**. На странице «Брифы» — селектор привязки брифа к проекту
   (`briefs.project_key`); на странице проекта — карточка с ответами брифа; `/api/dev/brief` отдаёт бриф по токену проекта
   (для локального дизайна/спеки). *TODO (мастер + реквизиты + чек-лист + доки)* — после Модулей 3/4.
3. **Гайды** (фаза 2) — ✅ **РЕАЛИЗОВАНО**. Библиотека инструкций (`guides`, markdown), раздел админки **«Гайды»** (CRUD),
   на странице проекта — тумблеры «Гайды клиенту» (`project_guides`), у клиента — раздел **«Подготовка»** (включённые гайды,
   аккордеоны). Сидированы стартовые: GitHub, Railway, TG-бот. Вместо «блоков внутри страницы» — гранулярные гайды (каждый блок = отдельный гайд).
4. **Договоры/счета**: `legal_entities` + `contract_templates` + подстановка реквизитов → PDF договор/НДА + счёт.
   *Требует от пользователя*: шаблоны договора/НДА, реквизиты ФОП/ТОВ, формат счёта.

## 6. Бриф: адаптивная форма (прогрессивное раскрытие)
- **Шаг 0 — тип проекта** (одна развилка): визитка · лендинг-услуга · интернет-магазин · SaaS · портфолио · другое.
- **Общий блок (всем, ≤5 полей):** что и для кого · есть ли бренд (цвета/лого/шрифты, аплоад) · референсы «нравится / НЕ нравится»
  (ссылки) · настроение 3 слова · язык(и).
- **Ветка-специфика** (только своя): визитка (блоки, форма заявки) · лендинг (оффер, целевое действие, доверие, цены) ·
  магазин (категории, оплата, доставка, CRM) · SaaS (роли, 3–5 сценариев, тарифы, интеграции, авторизация).
- Поля — селекты/чекбоксы/короткий текст; обязательное по минимуму. Хранение — структурированный `payload`.
- Референсы «НЕ нравится» → попадают в **AVOID** дизайн-кита.

## 7. Дизайн-кит / спек-кит (как стыкуется)
- **Дизайн-кит** (`.claude/skills/ui-ux-pro-max`): из брифа собираю запрос → `search.py --design-system` → залоченный
  арт-дирекшен (палитра-токены, шрифты, эффекты, AVOID). Постоянный AVOID: «ИИ-дефолт cream/serif/terracotta,
  AI purple/pink gradients, generic hero+testimonials+pricing». Артефакты — в `projects/_дизайн/<slug>/`.
- **Спек-кит** (`decomposeSpec`/kickoff): из брифа + арт-дирекшена собираю спеку → декомпозиция на задачи разрабу.

## 8. Нейминг
- Контейнер у клиента — **«Кабинет»**; список шагов — **«Подготовка»** (пункты: *заполнить* / *инструкция*).
- Админ-библиотека страниц — **«Гайды»**. Раздел лидов — **«Лиды»**.
