# PM-портал Lambertain

Встроенная в сайт админка/портал управления задачами (собственная БД Postgres), с веб-версией
и Telegram Mini App из одного кода. Доступ по ролям.

## Роли

| Роль | Доступ |
|------|--------|
| **admin** (Никита) | Первый экран **ПРОЕКТЫ** (дашборд загрузки разрабов); второй — ЗАДАЧИ (`/admin/tasks`); команда, скилы |
| **contributor** | Главная = задачи, **назначенные ему**, табами по проектам; клик → сразу в задачу; метки NEW (проект/задача/коммент) |
| **client** | Задачи своего проекта, создать задачу/вопрос |

Роль в портале берётся из связки `tg_links` (создаётся инвайтом) либо web-пароль = admin.
Один проект ведёт **один** разработчик (`meta.defaultAssignee`), но разработчик может вести
несколько проектов; назначение проектов разработчику — на `/admin/team`.

## Статусы, корзины и табы

Статусы — свободный текст. `statuses.ts` канонизирует любой
статус в **корзину** по ключевым словам: `notStarted | inProgress | review | rework | done | blocked`.
Канонические значения портала: `Open`, `In Progress`, `Review`, `Rework`, `Done`, `Blocked`.

**Жизненный цикл (переходы — автоматические, кроме финальной приёмки):**
- создана → `Open`;
- Claude разработчика **сам** ставит `In Progress` при старте и `Review` при завершении (`POST /api/dev/status`),
  при `review` передаёт `summary` — что сделано **простыми словами на языке задачи** → портал публикует это
  клиенту видимым комментарием (после модерации) и шлёт клиенту пуш. Это штатное завершение задачи;
  правила `summary` для дев-Клодов (в `dev-protocol.ts`): только правда по ЭТОЙ задаче (без обещаний файла/сборки, которых в ней нет),
  сдавать `review` ОДИН раз (повторный плодит дубли клиентских комментов), файлы клиенту прикладывать в задачу ДО review;
- эскалация вопроса клиенту (`/api/dev/escalate`) → `Blocked`; ответ клиента (`addTaskComment`) → снова `In Progress`;
- в `Review` **постановщик** (автор задачи) или админ проверяет результат: **принять** → `Done`, либо
  **на доработку** → `Rework` (с заметкой разработчику) — `ReviewActions` → `reviewTask`. Done ставит только постановщик.

Главная контрибьютора/клиента — **табы проект → статус** (`task-tabs.tsx`): В работе → Ревью → Доработка →
Готово → Не начатые → Заблок. Дефолт — «В работе»; если пусто — «Не начатые». В «Не начатые» клик = взять в работу.

## Зависимости (блокеры)

Задача может зависеть от других задач (`task_deps`, редактор на странице задачи). Пока хотя бы один
блокер не в корзине `done`, задача **автоматически** показывается в «Заблок.» (поверх своего статуса)
и её нельзя взять в работу. Блокеры закрыли → задача разблокирована на следующем рендере.

## Уведомления в Telegram (адресно по ролям)

Event-driven из server actions (`notify.ts`): tg_id резолвится из `tg_links` в момент отправки —
заработает, как только человек привязан (инвайт). Адресно:
- клиент написал коммент → ответственному разработчику + админу;
- команда ответила клиенту (видимый коммент) → клиенту/сотруднику проекта;
- внутренний коммент → ответственному разработчику; новая задача → исполнителю; «Готово» → клиенту.
- **Картинки в боте**: `/api/files/<id>` из текста/описания дослаются как фото (`sendPhoto`, байты из `attachments`).

## Поля проекта и видимость в карточке «Детали и доступы»

Карточка проекта (`project-info-card.tsx`) показывает поля по роли смотрящего; видимость каждого поля настраивается
в форме проекта (`meta-form.tsx`) двумя чекбоксами **«видно клиенту/сотруднику»** и **«видно разрабу»**
(`meta.fieldVisibility`, источник дефолтов — `field-visibility.ts`). Поля: prod/dev URL, дизайн, devInfo, спека, аккаунты.

- **Аккаунты входа** — два блока (логин/пароль/нотатка, добавляемые строки) под **Prod URL** и **Dev URL**
  (`meta.prodAccounts` / `meta.devAccounts`), каждый со своей видимостью.
- **Реестр кастомных полей** (`project-fields.ts`, пополняется кодом по запросу): соцсети/мессенджеры/доступы/аналитика.
  В настройках проекта — селект «Додати поле» → поле появляется с подполями + чекбоксами видимости; включённые
  хранятся в `meta.enabledFields` / `meta.customFields`. Постоянный `key` поля менять нельзя (по нему хранятся значения).
  Добавить новое поле (для нового клиента) = дописать запись в `PROJECT_FIELD_DEFS` — оно сразу в селекте всех проектов.
- **Railway/Vercel** — поля реестра с флагом `backed` (`clientDeploy`/`clientVercel`): UI вкл/выкл и видимость как у обычных,
  но значения по-прежнему хранятся в `meta.clientDeploy`/`meta.clientVercel` (форма проецирует их туда на save,
  карточка читает оттуда) — **логика деплоя/мониторинга не затронута**.

## Доставка dev → client (на странице проекта)

`deliver.ts` + панель (только при заданных dev/client git): squash-пуш состояния dev-репо одним коммитом
в client-репо в выбранную ветку (дефолтная — авто-деплой клиента; отдельная — клиент мержит сам), через GitHub API.
**Перед пушем — проверка схемы БД** (diff schema/migration-файлов dev↔client): если менялась —
портал **сам определяет авто-накат** (`detectMigrateOnDeploy`: ищет `prisma migrate deploy`/`drizzle-kit
migrate`/`migrate.mjs` и т.п. в `package.json` scripts start/build/release или в railway/nixpacks-конфиге репо).
Если деплой сам катит миграцию (`migratesOnDeploy`) — предупреждение зелёное и НЕ блокирует (миграция —
неотъемлемая часть деплоя). Иначе — красное, требует ручного подтверждения. Флаг `clientAutoMigrate` —
принудительный оверрайд на случай, если детект не сработал. Для проектов с `clientDeploy` (Railway-токен+IDs
в meta) — апрув клиентского деплоя и мониторинг статуса.

## Синхронизация client → dev (свежий клиентский код в наш дев-репо)

`sync-client.ts` + `POST /api/dev/sync` (авторизация токеном проекта). Обратный поток к доставке:
в **спільній розробці** над клиентскими репо коммитит ещё и разработчик клиента, а **прямого доступа
к клиентским репо у нашего разработчика нет** — их видит только портал (`GITHUB_TOKEN`, мы коллабораторы).
Эндпоинт затягивает свежие клиентские **интеграционные ветки** (дефолтная + `develop`, если есть — автодетект,
без ручных флагов) и публикует их в НАШ дев-репо отдельными ветками `client-sync/<branch>` (реальная git-история,
те же SHA; по всем парам репо проекта — основная `devGit`/`clientGit` + `extraRepos`). Дальше разработчик локально:
`git fetch origin` → `git merge`/`rebase` `origin/client-sync/<branch>` в рабочую ветку — **конфликты разрешает сам**,
портал в мерж не вмешивается. Вызов прописан в collaborative-протоколе (начало каждой сессии, до правок).
- **git-бинарь на сервере** (в отличие от `deliver.ts` на GitHub API): дев-репо — НЕ форки клиентских (нет общего
  object-store), перенести историю одним API-вызовом нельзя. Наличие git в рантайме гарантирует `nixpacks.toml`
  (`aptPkgs += git`). Токен — через `http.extraHeader` (не в URL/`.git/config`). Server-side, Node runtime.

## Claude Code разработчика ↔ портал (протокол + эскалация)

Claude Code разработчика работает с порталом по токену проекта автономно (протокол — в `CLAUDE.md` дев-репо
между маркерами `LAMBERTAIN-PROTOCOL`; единый источник текста — `dev-protocol.ts` `protocolBlock`).
- `GET /api/dev/tasks` `?id=` — требование + `tags` + тред + `awaitingClient`/`lastClientAnswer`; `GET /api/dev/skills?tags=` — плейбуки.
- `POST /api/dev/status` `{taskId, status:"in_progress"|"review", summary?}` — Claude сам ставит статусы; при `review` `summary` → клиенту.
- `POST /api/dev/escalate` `{taskId, question, kind?}` — вопрос клиенту (→ Blocked) или Никите (`kind:"admin"`).
- `POST /api/dev/handoff` `{taskId, action}` — задача требует **ops-шага владельца** (деплой/регистрация/токен): ставит `tasks.owner_action`,
  уведомляет супер-админа, **статус не меняет** (клиент видит «в работе» — флаг внутренний). Разработчик берёт следующую незаблокированную
  задачу (в списке `ownerAction != null` — пропускает). Владелец на странице задачи жмёт «Выполнил» (`markOwnerActionDone`) → задача идёт
  в Done (если autoDone) или Review. Эти задачи входят в 15-мин напоминание супер-админу (поллер `remindSuperAdmin`).

**Раскладка протокола (`protocol-deploy.ts`, единый источник):**
- **авто** при привязке нашего dev-репо к проекту (`saveMeta` с `devGit` → `after(layProtocol)`) — новые проекты/репо
  подхватываются сами;
- **кнопка «Обновить протокол в дев-репо»** на `/admin/projects` (`redistributeProtocol` → `layProtocolAll`) — когда
  поменялся текст протокола;
- только в НАШИ репо (`Lambertain/*`), не в клиентские; сам портал (`lambertain-site`) пропускается.
- Защита от утечки: при доставке dev→client (`deliver.ts`) **вырезается** `LAMBERTAIN-PROTOCOL`-блок из CLAUDE.md и
  выкидываются внутренние файлы (`.lambertain/`, `esc.json`, `.env*`) — токен и протокол клиенту НЕ попадают.

## Глобальный фидбек-проект (Lamb.dev)

Проект `DEV` «Lamb.dev» (`meta.feedback`, привязан к нашему репо) **виден всем** ролям автоматически (и новым
юзерам — через `scope.ts`). Каждый пишет туда пожелания по порталу и **видит только свои** задачи
(`feedback.ts` `mergeFeedback`: убирает чужие фидбек-задачи, подмешивает свои; админ — все). Фидбек идёт **без
апрува** (напрямую админу + уведомление). В табе проекта — intro-баннер (i18n), что это и что видно только своё.

Lamb.dev всегда идёт **последним** во всех списках (`byFeedbackLast` в `page.tsx`: табы, карточки) и **не дефолтом** в селекте
создания задачи (`ChatIntake` берёт первый НЕ-фидбек проект) — иначе клиент по невнимательности создаёт задачи своего проекта
в Lamb.dev (`listProjects ORDER BY key` ставил `DEV` раньше, напр., `GP`). У разработчика Lamb.dev — последняя карточка на главной.

## Клиентская приватность (агентство = Lambertain)

Клиент не должен видеть, что работу делают конкретные исполнители — для него это работа агентства:
- **Исполнитель скрыт**: имя assignee не показывается клиенту (список задач, карточка, дашборд — admin-only).
- **Автор комментария**: коммент хранит **реального автора** (`addComment(..., authorLogin)` → member по логину; бот/супер-админ без логина → `lambertain`). При выводе клиенту команда (разработчик/админ) маскируется под «Lambertain» (`comments-view`), а **его собственный коммент и комменты сотрудника — по имени**. Раньше автор не сохранялся вовсе → любой коммент клиента показывался как «Lambertain» (баг, исправлено).
- **Модерация клиент-видимых комментов** (`comments.approved`, `src/lib/moderation.ts`): ЛЮБОЙ клиент-видимый
  коммент от команды (разработчик/сотрудник/админ — **кроме супер-админа** и самого клиента) создаётся `approved=false`
  (pending): клиент его **не видит** и **не получает пуш**; уведомление уходит супер-админу (Никите) на модерацию.
  Касается всех каналов: ручной коммент в UI (`addTaskComment`, авто-полиш в агентский голос через `draftClientMessage`),
  вопрос Claude разработчика (`/api/dev/escalate kind:client`) и итог «готово до перевірки» (`/api/dev/status review`).
  Супер-админ на странице задачи: **Одобрить / Редактировать+одобрить / Не клиенту (внутр.)** (`moderateApprove/Edit/Reject`). «Не клиенту»
  (`rejectToInternal`) не удаляет — делает коммент **внутренним** (текст остаётся в треде команде, клиент не видит). Полное удаление — отдельная
  корзина (`superDeleteComment`, на любом коммента). Резолв любого вида убирает коммент из очереди напоминаний.
  На апруве коммент публикуется (`approved=true`) и клиент получает уведомление. Счётчики/`lastCommentAt` считают только
  одобренные. Прямой ИИ-ответ клиенту (`ClientReply`) оставлен **только супер-админу** (у разработчиков убран).
- **ИИ-проработка**: спецификация — внутренний коммент (клиент не видит); клиенту виден лишь уточняющий
  вопрос `ask_client` от Lambertain (без имён исполнителей).
- **Внутренние комментарии**: `comments.visibility` (`client`/`internal`). Команда по умолчанию пишет **внутренний**
  коммент (клиент не видит); галочка «Видно клиенту» — для клиентских ответов. ИИ-ревью кода — внутренний.
- **Ответ клиенту через ИИ** (`creply`/`draftReplyFromDevFeedback`): разработчик даёт суть своими словами →
  ИИ сверяет с задачей и кодом и формулирует ответ **от имени Lambertain** → разработчик утверждает/правит →
  публикуется клиенту. Так исполнитель не пишет клиенту «от себя».

## Команда: онбординг и перенос истории

Инвайт-ссылка (`/admin/team`): роль + **несколько проектов чекбоксами** (алфавит) или создание нового проекта
прямо в форме. Разработчик при входе по Telegram становится **ответственным** (`defaultAssignee`) на выбранных
проектах — сразу виден на дашборде (не нужно потом доназначать). Клиент/сотрудник — привязан к первому проекту.
Срок годности инвайта — **месяц** (`DEFAULT_TTL_HOURS=720` в `invites.ts`; дефолт `/api/admin/invite` тот же).

**Онбординг-уведомление при добавлении в проект** (`onboarding-notify.ts`, во всех путях: `redeemInvite`,
`approveAccess`, `saveUserProjects`): когда участника добавляют в проект, где уже есть задачи —
разработчику шлём, сколько задач в работе; клиенту/сотруднику — что по проекту **уже выполнено**
(иначе он не увидит работу, закрытую до его добавления). В `saveUserProjects` — только по НОВЫМ проектам
(diff через `getUserProjectKeys`). Предпросмотр без отправки: `GET /api/admin/onboarding-preview?projectKey=&role=`.

На странице проекта (`/admin/projects/[key]`) — панель **«Пользователи проекта»** (`project-users.tsx`): список
участников этого проекта (клиент, сотрудники, ответственный разработчик) без фильтров; убрать из проекта
(не удаляя из портала) / удалить пользователя; добавить уже присоединившегося или сгенерировать инвайт сразу
в этот проект. Та же логика проектов, что на `/admin/team` (member_projects ∪ tg_links.project_key, разраб по `defaultAssignee`).

Уход с YouTrack: ники YouTrack удаляются из `members`, но авторство сохраняется — `comments.orig_author_login/role`,
`tasks.orig_assignee_login/orig_reporter_login` (штампуются миграцией). Блок «Перенос истории» на `/admin/team`
показывает осиротевшие логины и привязывает их историю к новому tg-пользователю (`relinkMember`).

## Вложения / картинки задач

Картинки задач **самохостим**: таблица `attachments` (bytea), роут `/api/files/[id]` отдаёт байты
(только авторизованным), в описании — `![](/api/files/<id>)`. Описание рендерится как markdown
(`markdown.tsx`, react-markdown + remark-gfm; есть стили таблиц/цитат + горизонтальный скролл таблиц на узких экранах).

**Панель форматирования** (`markdown-toolbar.tsx`) — общий тулбар над полем ввода в форме создания задачи
(`chat-intake.tsx`) и в комментариях (`comment-box.tsx`): кнопки B/I/код/заголовок/списки/ссылка оборачивают
выделение в Markdown, «вставить таблицу» (строк×столбцов) кладёт GFM-заготовку. Без сырого HTML и доп. зависимостей.

## Дашборд загрузки (главная админа)

`dev-dashboard.tsx`: проекты сгруппированы по ответственному разработчику. На каждого — суммарная
стоимость его проектов и карточки: стоимость, дней ведётся (`today − startedAt`), до конца
(`deadline − today`), **два прогресс-бара** (по времени до дедлайна и по задачам done/total), счётчик
задач. Экономика проекта — поля `ProjectMeta` (jsonb, без миграции схемы): `cost`, `currency` (деф. `₴`),
`startedAt`, `deadline`. Проекты без ответственного — блок «Без ответственного».

## Постановка задачи: запрос → фоновая ИИ-проработка

Клиент/сотрудник/админ описывает запрос в окне «Новая задача» (`chat-intake.tsx`): лента сообщений —
текст, скрины и файлы в произвольной хронологии (скрин/файл уходит в ленту **сразу** отдельным
сообщением, текст — по кнопке отправки; голосовой ввод — зажать кнопку). Отправленный текст
**редактируется/удаляется**. Когда юзер закончил — кнопка «Создать задачу».

- Задача создаётся **сразу** из сырого запроса: описание собирается из блоков ленты с сохранением
  порядка (текст → скрин → текст → скрин), картинки самохостятся (`appendRequestBlocks`, `attachments`).
  Заголовок — первая строка запроса как есть. Задача **сразу назначается разработчику проекта** (`defaultAssignee`) + пуш ему.
  > Отправка идёт **fetch-ом на роут `POST /api/portal/create-task`** (session-cookie), НЕ Server Action:
  > ссылка на Server Action инвалидируется при деплое и Next форсит перезагрузку (клиент терял загруженные
  > скрины). URL роута деплой переживает, поэтому выкатки никак не мешают создавать задачу. Логика — общая,
  > в `src/lib/task-intake.ts` (`createRequestTaskCore`), её же дёргает тонкий Server Action `createRequestTask`.
  > Вложения (base64) большие → в `next.config.ts` поднят `experimental.serverActions.bodySizeLimit = "15mb"`
  > (для оставшихся Server Actions; у роута лимита body нет).
- **Апрув-гейт**: задача сотрудника требует утверждения (`approval_status='pending'`: проект с клиентом →
  утверждает клиент, без клиента → админ). Пока pending — разработчику задачу **НЕ отдаём**; утверждающий может
  отредактировать задачу (`TaskEdit`, admin: заголовок/текст/исполнитель/приоритет → `editTask`). После `approved`
  (`setApproval`) задача назначается разработчику + пуш (`assignProjectDevAndNotify`). Без апрува (клиент/админ-создатель) —
  назначается сразу при создании.
- **ИИ-триажа на портале НЕТ** (убран как лишняя интеграция без контекста репо). Весь разбор делает Claude
  разработчика локально (полный доступ к репо): читает задачу и переписку → смотрит код → если что-то реально
  непонятно, задаёт клиенту уточнение комментом (`/api/dev/escalate`, через модерацию Никиты) → реализует.
  Назначение+пуш разработчику делает `assignProjectDevAndNotify` (в `task-intake.ts`) сразу при создании/после апрува.
  Теги `{type, complexity, skills}` проставляются только при kickoff (декомпозер спеки); у обычных задач пусты —
  сложность/скилы оценивает сам Claude разработчика. Dev-API: `GET /api/dev/tasks?id=` отдаёт задачу+переписку+`projectSpec`.
- Ручная форма «текст → структура» (`structurer.ts`, `structureDraft`/`createFromDraft`) сохранена отдельно.

## Онбординг-инструкция клиента (внутренняя, интерактивная)

Для подключения нового клиента (GitHub-репо + хостинг Railway + токен) — пошаговая инструкция:
- **Редактор** (`/admin/onboarding`, только админ): шаги (заголовок + тело Markdown), добавление/удаление/
  порядок, **вставка скринов** (загрузка → `![](/api/onboarding-media/<id>)` в позицию курсора), и на каждом шаге —
  выбор **собираемого поля** (`collect`: нет / репозиторий / Railway-токен). Хранение: `settings` (JSONB ключ
  `onboarding`, сид uk-текста миграцией, ON CONFLICT DO NOTHING) + `onboarding_media` (bytea).
- **Страница `/onboarding`** — внутренняя (требует входа). Аккордеон Крок 1/2/3 с прогрессивным раскрытием
  (следующий после «Виконано», прогресс в localStorage). На шагах с `collect` клиент **вводит данные** (ссылка
  на репо, токен) — они сохраняются в поля проекта (`meta.clientGit`, `meta.clientDeploy.railwayToken`) через
  `submitOnboardingField`. Когда все поля заполнены — снимается флаг показа.
- **Инвайт** (`/admin/team`): для клиента — чекбокс «Показати інструкцію при вході» (`invites.show_onboarding`).
  При redeem ставит `project.meta.showOnboarding=true`. Тогда на главной клиента — баннер со ссылкой на `/onboarding`.

## Наборы инструкций (публичные ссылки)

Блоки-гайды (`/admin/guides`) можно собирать в **наборы** и отдавать публичной ссылкой — лидам/клиентам в любой
мессенджер, без портала и без привязки к человеку.
- Таблица `instruction_sets` (token, title, `guide_ids INT[]` — порядок блоков). UI — внизу `/admin/guides`
  («Набори інструкцій»): отмечаешь блоки галочками (нумеруются в порядке выбора) → «Створити посилання».
- Публичная страница `/i/<token>` (без авторизации, `robots noindex`) рендерит выбранные гайды по порядку.
  Картинки гайдов (`/api/guide-files/<id>`) сделаны публичными — нужны на этой странице.
- `getInstructionSetByToken` отдаёт гайды в порядке `guide_ids`. Тот же набор привязывается к инвайту (селект
  «Набір інструкцій при вході» вместо простой галочки) → при redeem пишется `meta.onboardingSetToken`, баннер на главной клиента ведёт на `/i/<token>`.
- **Гайды мультилокальные** (uk основная + `title_ru/body_ru/title_en/body_en`, fallback на uk через `guideText`). Редактор —
  табы UA/RU/EN; клиент видит на своей локали; на `/i/<token>` — переключатель `?lang=`.

## Бриф (нулевая стадия: лид → Telegram → ответы)

Бриф — стадия до клиента/цены/проекта. Контакт лида **не спрашивается формой** — он определяется по Telegram при авторизации.
- `/admin/briefs` («Брифы»): «Новый лид» (метку можно не указывать) → `createBrief` → ссылка `briefLink(token)` =
  Mini App диплинк `https://t.me/<bot>/<app>?startapp=brief-<token>`. Отправляешь лиду — он жмёт «Открыть».
- Бот открывает Mini App `/tma`; `start_param` вида `brief-<token>` → `/tma` редиректит на `/brief/<token>` (в TMA-контексте,
  страница подключает `telegram-web-app.js`). Лид заполняет бриф; форма читает `window.Telegram.WebApp.initData`.
- `submitBriefAction` валидирует initData (`validateInitData`) → `submitBrief(token, type, payload, tg)` пишет
  `tg_id/tg_username/tg_name`; `label` подставляется из названия компании, иначе из tg-имени/`@username`. Ответы — в `/admin/briefs`
  (с tg-контактом в строке) и в бот админу. Поля формы общие: компания (обяз.), что нужно, бюджет, дедлайн — **без поля контактов**.
- Привязка брифа к проекту — селект в строке (`linkBrief`): ответы становятся доступны проекту и Claude через токен.
- **Формат заполнения — A/B на первом экране** (`brief-intake.tsx`): лид выбирает «Форма» (всё на одном экране) или
  «Чат» (диалог по одному вопросу, без LLM — детерминированный сценарий). Вопросы у обоих общие — из единого
  `brief-schema.ts` (источник правды: `TYPES/COMMON/BRANCH/fieldsFor`); добавил поле туда → появляется и в форме, и в чате.
  Выбранный формат пишется в `payload._mode` (`form|chat`, без миграции); в `/admin/briefs` — бейдж режима и счётчик
  «A/B заполнений: форма N · чат M», чтобы видеть, что в спросе.

## Секреты проекта + умный handoff (self/client/owner)

- **Секреты** (`project_secrets`, страница «Секрети та доступи» на проекте, только админ — страница `requireAdmin`):
  токены/логины/ключи. Разработчик-человек их НЕ видит; его Claude-код читает через `GET /api/dev/secrets` (по токену проекта,
  инструкция в протоколе). Старые `meta.credentials`/`clientDeploy.railwayToken` перенесены миграцией.
- **Handoff** (`POST /api/dev/handoff`): портал-Клод (`classifyHandoff`, Anthropic) определяет адресата ops-шага:
  - `self` — разработчик делает сам (флаги не ставим, internal-след);
  - `client` — действие клиента (регистрация сервиса/доступ): `client_action` на задаче + коммент + **пуш клиенту в Telegram**
    («🔑 Потрібно зареєструвати ‹X›») + блок `ClientActionBar` (гайд + поле данных + «Готово»). «Готово» (`markClientActionDone`) →
    данные в `project_secrets` (`filled_by='client'`) → задача назад разработчику (уведомление). Гайда нет → `generateGuide`
    (AI, 3 локали) создаёт его на лету и привязывает (`client_action_guide`).
  - `owner` — инфра агентства → владельцу (как раньше, `owner_action`).
  - dev API скрывает из очереди задачи с `ownerAction`/`clientAction` (разработчик их пропускает).

## Договоры (генерация по шаблонам)

Экран `/admin/contracts` (только админ) — сборка договоров из шаблонов с плейсхолдерами `{{key}}`.
Три вкладки: **Договори** (список + просмотр/печать), **Новий договір** (мастер), **Виконавці (ФОПи)**, **Шаблони**.

- **Модель данных** (`db.ts` + `scripts/migrate.mjs`): `contractors` (наши ФОПы-исполнители: name/address/ipn/iban/
  bank_name/bank_mfo/bank_edrpou/phone), `contract_templates` (title/lang/body), `contracts` (number/contract_date/
  city/title/template_id/contractor_id/client_requisites/vars JSONB/body — body это снимок отрендеренного текста).
- **Плейсхолдеры** (`src/lib/contracts.ts`, чистая логика): `{{contractor.<field>}}` подставляются из выбранного ФОПа;
  `{{client.requisites}}` — поле «Реквізити Замовника» (вставка блока реквизитов клиента); `{{payments}}` — конструктор
  графика платежей; `{{number}}/{{date}}/{{city}}` — реквизиты договора (`date` форматируется в укр. вид `formatUaDate` →
  «15» червня 2026 р.); любые прочие `{{поле}}` (`subject`, `price`, `term`…) автоматически становятся полями в мастере
  (`dynamicPlaceholders`). Подписи — `FIELD_LABELS_UK`, многострочные — `MULTILINE_KEYS`.
- **График платежей** (`{{payments}}`, `PaymentsBuilder`): строки «сумма + условие», где условие свободное — срок ИЛИ
  этап разработки (первый платёж по умолчанию помечен предоплатой). `buildPaymentsText` собирает «Платіж № 1: 10 000 грн — …;»
  построчно; `paymentsSum` сверяет сумму графика с общей стоимостью (`{{price}}`) и подсвечивает расхождение.
- **Создание**: `createContractAction` рендерит `renderContract(template, ctx)` → сохраняет снимок `body`. Просмотр
  `/admin/contracts/[id]` — белый «лист» (serif), `@media print` прячет навигацию → печать или сохранение в PDF браузером.
- **Сид** (миграция, идемпотентно): типовой договор «Послуги веб-розробки (ФОП)» (укр) + один ФОП-исполнитель, если
  справочник пуст. Дальше всё правится через UI.

## Скилы интейка (плейбуки)

Интейк подбирает под тип задачи **скил** — плейбук с инструкциями. Работает по модели
progressive disclosure (как Agent Skills у Anthropic): в системный промпт идёт только
`slug: заголовок [триггеры]` (Level 1), а полный текст подгружается по требованию через
инструмент `use_skill` (Level 2). Если подходящего скила нет — интейк создаёт новый
(`create_skill`, помечается «авто», админу уходит уведомление).

- Хранилище: таблица `skills` (`slug, title, triggers, playbook, auto_generated`).
- Стартовый набор — **реальные полные скилы** из открытых источников (`SKILL.md`),
  лежат в `scripts/skills/*.md`, сидятся миграцией `scripts/migrate.mjs`:

  | slug | источник |
  |---|---|
  | ui-ux-pro-max | nextlevelbuilder/ui-ux-pro-max-skill (основной дизайн) |
  | frontend-design | anthropics/skills |
  | web-design-guidelines | vercel-labs/web-interface-guidelines |
  | react-best-practices | vercel-labs/agent-skills |
  | systematic-debugging | obra/superpowers |
  | tdd | obra/superpowers (test-driven-development) |
  | code-review | obra/superpowers (requesting-code-review) |
  | verification | obra/superpowers (verification-before-completion) |
  | writing-plans | obra/superpowers |
  | claude-api | anthropics/skills |
  | postgres | supabase/agent-skills |
  | webapp-testing | anthropics/skills |

- Добавить/обновить скил: положить `scripts/skills/<slug>.md` (формат SKILL.md, фронтматтер
  срезается при сиде) и дописать строку в `SEED_SKILLS` (`migrate.mjs`). Сид идемпотентен;
  авто-скилы (`auto_generated=true`) миграция не затирает.
- UI: `/admin/skills` — список (превью со «свернуть/развернуть») + расход токенов.

## Локальный запуск

```
npm install
npm run dev                 # localhost:3000/admin
node --env-file=.env.local scripts/poller.mjs   # поллер (DRY_RUN=1 POLL_ONCE=1 для теста)
```

Postgres локально: `localhost:5434`, БД `lambertain_pm` (создаётся автоматически схемой).

## Деплой на Railway

1. Сервис **web** из репозитория: build `npm run build`, start `npm start`.
2. Плагин **Postgres** → `DATABASE_URL` в web-сервис.
3. Второй сервис **poller** из того же репозитория: start `npm run poller`, те же env + `DATABASE_URL`.
4. Прописать все env в обоих сервисах.
5. В **@BotFather**: `/newapp` для @LambDev_bot → URL мини-аппы `https://<домен>/tma`,
   затем заполнить `TELEGRAM_MINIAPP_SHORTNAME`.

Роли пользователей берутся из инвайта (`/admin/team`), а не из внешней системы.
