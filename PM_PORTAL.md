# PM-портал Lambertain

Встроенная в сайт админка/портал управления задачами (собственная БД Postgres), с веб-версией
и Telegram Mini App из одного кода. Доступ по ролям.

## Роли

| Роль | Доступ |
|------|--------|
| **admin** (Никита) | Первый экран **ПРОЕКТЫ** (дашборд загрузки разрабов); второй — ЗАДАЧИ (`/admin/tasks`); команда, скилы |
| **contributor** | Главная = задачи, **назначенные ему**, табами по проектам; клик → сразу в задачу; метки NEW (проект/задача/коммент) |
| **client** | Задачи своего проекта, создать задачу/вопрос |

Роль в портале берётся из связки `tg_links` (создаётся инвайтом) либо web-пароль = admin.
Один проект ведёт **один** разработчик (`meta.defaultAssignee`), но разработчик может вести
несколько проектов; назначение проектов разработчику — на `/admin/team`.

## Статусы, корзины и табы

Статусы — свободный текст (устойчиво к импорту из YouTrack). `statuses.ts` канонизирует любой
статус в **корзину** по ключевым словам: `notStarted | inProgress | review | done | blocked`.
Канонические значения, которые ставит портал: `Open`, `In Progress`, `Review`, `Done`, `Blocked`.

Главная контрибьютора/клиента — **табы проект → статус** (`task-tabs.tsx`). Порядок табов:
В работе → Ревью → Готово → Не начатые → Заблок. Дефолт — «В работе»; если там пусто, открываются
«Не начатые». В табе «Не начатые» клик по задаче = взять в работу (статус → In Progress).

## Зависимости (блокеры)

Задача может зависеть от других задач (`task_deps`, редактор на странице задачи). Пока хотя бы один
блокер не в корзине `done`, задача **автоматически** показывается в «Заблок.» (поверх своего статуса)
и её нельзя взять в работу. Блокеры закрыли → задача разблокирована на следующем рендере.

## ИИ код-ревью (on-demand)

Кнопка «ИИ-ревью кода» на странице задачи — **только админ** (разработчики ревьюят через свой Claude + глазами).
`review.ts`: Claude читает дифф по ссылке
(`review_ref`: PR/коммит/ветка) либо HEAD dev-репо и пишет вердикт **комментарием**. Статус НЕ меняет —
это вторая пара глаз по диффу, рантайм разработчик проверяет сам на dev-хостинге. Автозапуска нет
(токены тратятся только по кнопке). Расход пишется в `token_usage` (kind=`review`).

## Уведомления в Telegram (адресно по ролям)

Event-driven из server actions (`notify.ts`): tg_id резолвится из `tg_links` в момент отправки —
заработает, как только человек привязан (инвайт). Адресно:
- клиент написал коммент → ответственному разработчику + админу;
- команда ответила клиенту (видимый коммент) → клиенту/сотруднику проекта;
- внутренний коммент → ответственному разработчику; новая задача → исполнителю; «Готово» → клиенту.
- **Картинки в боте**: `/api/files/<id>` из текста/описания дослаются как фото (`sendPhoto`, байты из `attachments`).

## Доставка dev → client (на странице проекта)

`deliver.ts` + панель (только при заданных dev/client git): squash-пуш состояния dev-репо одним коммитом
в client-репо в выбранную ветку (дефолтная — авто-деплой клиента; отдельная — клиент мержит сам), через GitHub API.
**Перед пушем — проверка схемы БД** (diff schema/migration-файлов dev↔client): если менялась, требует
подтверждения (миграцию на клиентскую БД накатываешь вручную). Для проектов с `clientDeploy` (Railway-токен+IDs
в meta) — апрув клиентского деплоя и мониторинг статуса.

## Claude Code разработчика ↔ портал (протокол + эскалация)

Чтобы джун не пересылал вопросы скринами, Claude Code разработчика работает с порталом по токену проекта
автономно (инструкция — в `CLAUDE.md` дев-репо, читается при старте сессии; раскладка — `scripts/distribute-protocol.mjs`).
- `GET /api/dev/tasks` — задачи; `?id=` — спека + тред комментов + `awaitingClient` / `lastClientAnswer`.
- `POST /api/dev/escalate` `{taskId, question, kind?}` — Claude эскалирует вопрос: `client` (ИИ оформляет вопрос
  клиенту от лица агентства → клиент-видимый коммент с маркером + уведомление клиенту); `admin` (внутренний + Никите).
- Claude технические развилки решает сам по спеке/конвенциям; ответ клиента подтягивает, перечитывая задачу
  (`awaitingClient`/тред) — без ручного relay. `dev-protocol.ts` — генератор протокол-блока + маркер эскалации.

## Клиентская приватность (агентство = Lambertain)

Клиент не должен видеть, что работу делают конкретные исполнители — для него это работа агентства:
- **Исполнитель скрыт**: имя assignee не показывается клиенту (список задач, карточка, дашборд — admin-only).
- **Автор комментария** для клиента всегда «Lambertain» (не имя работника); чужой репортёр — тоже скрыт/Lambertain.
- **ИИ-проработка**: спецификация — внутренний коммент (клиент не видит); клиенту виден лишь уточняющий
  вопрос `ask_client` от Lambertain (без имён исполнителей).
- **Внутренние комментарии**: `comments.visibility` (`client`/`internal`). Команда по умолчанию пишет **внутренний**
  коммент (клиент не видит); галочка «Видно клиенту» — для клиентских ответов. ИИ-ревью кода — внутренний.
- **Ответ клиенту через ИИ** (`creply`/`draftReplyFromDevFeedback`): разработчик даёт суть своими словами →
  ИИ сверяет с задачей и кодом и формулирует ответ **от имени Lambertain** → разработчик утверждает/правит →
  публикуется клиенту. Так исполнитель не пишет клиенту «от себя».

## Команда: онбординг и перенос истории

Инвайт-ссылка (`/admin/team`): роль + **несколько проектов чекбоксами** (алфавит) или создание нового проекта
прямо в форме. Разработчик при входе по Telegram становится **ответственным** (`defaultAssignee`) на выбранных
проектах — сразу виден на дашборде (не нужно потом доназначать). Клиент/сотрудник — привязан к первому проекту.

Уход с YouTrack: ники YouTrack удаляются из `members`, но авторство сохраняется — `comments.orig_author_login/role`,
`tasks.orig_assignee_login/orig_reporter_login` (штампуются миграцией). Блок «Перенос истории» на `/admin/team`
показывает осиротевшие логины и привязывает их историю к новому tg-пользователю (`relinkMember`).

## Вложения / картинки задач

Импорт из YouTrack сохранял только текст описания, где картинка — это имя вложения (`![](image.png)`),
а файл лежал по истекающему подписанному URL YouTrack. Поэтому картинки **самохостим**: таблица
`attachments` (bytea), роут `/api/files/[id]` отдаёт байты (только авторизованным). Бэкофилл —
`scripts/import-attachments.mjs`: качает вложения из YouTrack и переписывает в описании
`![](имя)` → `![](/api/files/<id>)`. Описание рендерится как markdown (`markdown.tsx`, react-markdown).

## Дашборд загрузки (главная админа)

`dev-dashboard.tsx`: проекты сгруппированы по ответственному разработчику. На каждого — суммарная
стоимость его проектов и карточки: стоимость, дней ведётся (`today − startedAt`), до конца
(`deadline − today`), **два прогресс-бара** (по времени до дедлайна и по задачам done/total), счётчик
задач. Экономика проекта — поля `ProjectMeta` (jsonb, без миграции схемы): `cost`, `currency` (деф. `₴`),
`startedAt`, `deadline`. Проекты без ответственного — блок «Без ответственного».

## Постановка задачи: запрос → фоновая ИИ-проработка

Клиент/сотрудник/админ описывает запрос в окне «Новая задача» (`chat-intake.tsx`): лента сообщений —
текст, скрины и файлы в произвольной хронологии (скрин/файл уходит в ленту **сразу** отдельным
сообщением, текст — по кнопке отправки; голосовой ввод — зажать кнопку). Отправленный текст
**редактируется/удаляется**. Когда юзер закончил — кнопка «Создать задачу».

- Задача создаётся **сразу** из сырого запроса: описание собирается из блоков ленты с сохранением
  порядка (текст → скрин → текст → скрин), картинки самохостятся (`appendRequestBlocks`, `attachments`).
  Заголовок — обрезка первой строки (ИИ уточнит). Исполнитель пока не назначается. `tasks.ai_status='pending'`.
- Дальше — **фоновая ИИ-проработка** (`drafter.ts`, запуск через `after()`; страховка — `ai_status='pending'`).
  Agentic-цикл (те же скилы + чтение репо через GitHub) готовит **техническую спецификацию для разработчика**
  и пишет её **внутренним комментарием** (`visibility=internal`, клиент не видит) → назначает ответственного
  (`defaultAssignee`) → уведомляет разработчика. Только теперь задача «доступна» разработчику.
- Если без уточнения нельзя — `ask_client`: один вопрос по делу **клиентским комментарием** от Lambertain
  + уведомление клиенту, `ai_status='waiting'`. Ответ клиента (`addTaskComment`) возобновляет проработку.
- Общение модели — **только через инструменты** (`write_spec` / `ask_client`); свободный текст отбрасывается
  (не зависает и не вываливает рассуждения). Клиент на странице задачи видит баннер статуса проработки.
- Юзер может закрыть портал — результат (готовая задача или вопрос) придёт **уведомлением в бот**.
  Ручная форма «текст → структура» (`structurer.ts`, `structureDraft`/`createFromDraft`) сохранена отдельно.

## Скилы интейка (плейбуки)

Интейк подбирает под тип задачи **скил** — плейбук с инструкциями. Работает по модели
progressive disclosure (как Agent Skills у Anthropic): в системный промпт идёт только
`slug: заголовок [триггеры]` (Level 1), а полный текст подгружается по требованию через
инструмент `use_skill` (Level 2). Если подходящего скила нет — интейк создаёт новый
(`create_skill`, помечается «авто», админу уходит уведомление).

- Хранилище: таблица `skills` (`slug, title, triggers, playbook, auto_generated`).
- Стартовый набор — **реальные полные скилы** из открытых источников (`SKILL.md`),
  лежат в `scripts/skills/*.md`, сидятся миграцией `scripts/migrate.mjs`:

  | slug | источник |
  |---|---|
  | ui-ux-pro-max | nextlevelbuilder/ui-ux-pro-max-skill (основной дизайн) |
  | frontend-design | anthropics/skills |
  | web-design-guidelines | vercel-labs/web-interface-guidelines |
  | react-best-practices | vercel-labs/agent-skills |
  | systematic-debugging | obra/superpowers |
  | tdd | obra/superpowers (test-driven-development) |
  | code-review | obra/superpowers (requesting-code-review) |
  | verification | obra/superpowers (verification-before-completion) |
  | writing-plans | obra/superpowers |
  | claude-api | anthropics/skills |
  | postgres | supabase/agent-skills |
  | webapp-testing | anthropics/skills |

- Добавить/обновить скил: положить `scripts/skills/<slug>.md` (формат SKILL.md, фронтматтер
  срезается при сиде) и дописать строку в `SEED_SKILLS` (`migrate.mjs`). Сид идемпотентен;
  авто-скилы (`auto_generated=true`) миграция не затирает.
- UI: `/admin/skills` — список (превью со «свернуть/развернуть») + расход токенов.

## Локальный запуск

```
npm install
npm run dev                 # localhost:3000/admin
node --env-file=.env.local scripts/poller.mjs   # поллер (DRY_RUN=1 POLL_ONCE=1 для теста)
```

Postgres локально: `localhost:5434`, БД `lambertain_pm` (создаётся автоматически схемой).

## Деплой на Railway

1. Сервис **web** из репозитория: build `npm run build`, start `npm start`.
2. Плагин **Postgres** → `DATABASE_URL` в web-сервис.
3. Второй сервис **poller** из того же репозитория: start `npm run poller`, те же env + `DATABASE_URL`.
4. Прописать все env в обоих сервисах.
5. В **@BotFather**: `/newapp` для @LambDev_bot → URL мини-аппы `https://<домен>/tma`,
   затем заполнить `TELEGRAM_MINIAPP_SHORTNAME`.

## Известная настройка: роли в YouTrack

Поллер и экран «Клиенты» определяют клиента по роли **«Клиент»** в YouTrack.
Сейчас у пользователей роли не назначены — назначить в YouTrack
(Администрирование → проект → команда/роли), иначе клиентские триггеры не сработают.
Доступ в портал/Mini App от этого не зависит — там роль из инвайта.
```
