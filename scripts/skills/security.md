# Безопасность (security review)

Цель: не пропустить уязвимости и дыры; релевантные находки — в постановку задачи.

## Чек-лист (OWASP-ориентированный)
- **Аутентификация/сессии**: надёжное хранение сессий, истечение, защита кук (HttpOnly/SameSite/Secure), отсутствие предсказуемых токенов.
- **Авторизация (доступы)**: проверка прав на КАЖДОМ защищённом действии (не только в UI). Нет IDOR (доступ к чужому объекту по id). Серверная проверка роли.
- **Инъекции**: SQL — только параметризованные запросы; нет конкатенации ввода. Команды/шелл — без интерполяции. NoSQL/LDAP/шаблоны — экранирование.
- **XSS**: вывод пользовательского ввода экранируется; нет `dangerouslySetInnerHTML`/innerHTML с непроверенным контентом; CSP по возможности.
- **CSRF**: для cookie-сессий — токены/SameSite; мутации не по GET.
- **Валидация ввода**: на сервере, по белому списку; лимиты длины/типа; загрузка файлов — тип/размер/имя.
- **Секреты**: нет токенов/паролей в коде и в git; в .env/секретах; .gitignore покрывает .env*.
- **Данные**: минимизация выдачи (не отдавать лишние поля), маскирование чувствительного, шифрование где нужно, корректные CORS.
- **Зависимости**: известные уязвимые версии (audit), не тянуть лишнее.
- **Rate limiting / abuse**: защита логина/форм/вебхуков; проверка подписи вебхуков.
- **Ошибки**: не утекают стек-трейсы/детали БД клиенту.

## Итог
Найденные дыры — с точным местом и способом фикса — в задачу. Критичные (auth/доступы/инъекции) — приоритетом.
